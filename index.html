I designed a relational database (SQL Server) for a Paralympic-style event with tables for Countries, Athletes, Events, Event_Results, Medals, Tickets, and Volunteers. Every table has a clean primary key and foreign keys to enforce referential integrity, so invalid references (e.g., an athlete tied to a non-existent country) are impossible.

Data modeling followed 3NF: entity tables (Countries, Athletes, Events, Venues) are separated from transactional tables (Event_Results, Tickets, Volunteer_Assignments). Surrogate keys (INT IDENTITY) are used for stability, while natural keys (e.g., ISO country codes) are constrained as UNIQUE for fast lookups. Many-to-many relationships (e.g., Volunteers ‚Üî Events) are handled via junction tables with composite keys.

To make analytics self-serve, I created views such as v_MedalWinners (fast medal rollups by event/country/classification) and v_TicketSales (daily revenue by event/venue). These views pre-join the minimum set of tables and expose consistent column names for BI tools, so analysts don‚Äôt have to write complex joins repeatedly.

For governance and traceability, I added an audit table with an AFTER UPDATE/INSERT trigger on Event_Results and Tickets that captures who changed what and when (old/new values, user, timestamp). This enables post-event reconciliation and supports compliance requirements without granting broad write access.

I implemented a parameterized stored procedure (e.g., dbo.GetAthletesByCountryOrEvent) that returns athletes filtered by @CountryId and/or @EventId, with safe defaults and input validation. Common app/report flows call this SP instead of crafting raw SQL, improving reuse, plan caching, and security.

Performance was addressed with targeted nonclustered indexes on high-selectivity columns (Event_Results.EventId, Tickets.EventId, Athletes.CountryId). Covering indexes support frequent BI queries, while partitioning by EventDate can be enabled for very large seasons. Query plans were checked to avoid key lookups and ensure SARGable predicates.

Quality and consistency rules are enforced with CHECK constraints (valid medal types, non-negative ticket quantities/price), NOT NULL on required fields, and cascaded updates/deletes only where domain rules allow them. Reference data (countries, venues, classifications) is loaded via seed scripts and protected by limited DML privileges.

Security follows least-privilege: application logins map to a read-only role for views and SPs; write permissions are restricted to service accounts. PII (if stored) is masked in views, and access is mediated through stored procedures. Backups use full + differential + log strategy; a lightweight migration script (DDL + seed) initializes new environments.

Outcome: a production-style schema that answers real operator questions (‚Äúwho won yesterday?‚Äù, ‚Äúwhat was revenue per event/venue?‚Äù, ‚Äúwhich volunteer covered which session?‚Äù) with fast, reliable queries. The combination of views, stored procedures, indexing, and auditing demonstrates practical data engineering for transactional + reporting workloads.
